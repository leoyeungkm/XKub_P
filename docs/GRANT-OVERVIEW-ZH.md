## 三、核心機制詳解

### 3.1 合成永續 + XPLP 流動性金庫(GMX v1 模式)

沒有訂單簿。所有交易者的對手方是一個 KUSDT 流動性池(**XPLP**,類同 GMX 的 GLP / Hyperliquid 的 HLP / Jupiter 的 JLP):

- LP 存入 KUSDT 按當前淨值(NAV)鑄造 XPLP 份額;提取時按 NAV 銷毀換回 KUSDT
- **收入來源**:交易者虧損 + 全部手續費(開/平倉費 LP 份額、持倉借貸費、清算罰金、急速平倉費)流入池中推升 NAV
- **支出**:交易者盈利由池支付,NAV 下降——LP 是「莊家」,賺取統計優勢與費用,承擔做市風險
- **儲備係數 50%**:池必須保留至少未平倉名義值 50% 的儲備,LP 提款不可觸碰該儲備,確保交易者盈利有錢可付
- **提款冷靜期 15 分鐘**:每次存款後重新計時(按金額加權,防止小額存款重置大額計時的騷擾攻擊),阻止「見到大單即將獲利前搶先撤資」的三明治行為
- **死份額(MINIMUM_LIQUIDITY)**:首次存款鑄造時銷毀最小份額,杜絕經典的「首存通脹攻擊」(攻擊者以 1 wei 首存 + 直接轉帳操縱份額價格)

### 3.2 帳戶模型:交易餘額 + 代理密鑰

**交易餘額(Trading Balance)**:用戶將 KUSDT 存入 Router 託管,交易保證金只從這裡扣取,**永不觸碰用戶錢包中的其他資產**;餘額隨時可由用戶本人提回錢包。

**代理密鑰(Agent Key)**——與 Hyperliquid 的 Agent/API Wallet 同款模型:
- 瀏覽器本地生成、保存於 localStorage 的獨立密鑰;主錢包私鑰**永不接觸網頁**
- 合約層面權限受限:**只能替所有者下單/平倉,無法提款、無法轉移資金**(Router 逐函數強制,平倉款只會回到 owner 名下的交易餘額)
- **遺失無害**(清除瀏覽資料 / 換裝置):資金記名於主錢包,不受任何影響;重新授權一枚新代理(一筆交易)即恢復交易
- **洩漏可控**:最壞後果是「有人替你亂交易」而非盜走資金;可隨時 `setAgent(舊鑰, false)` 一筆撤銷;配合 300% 盈利上限,限制透過對手帳戶對賭抽取的理論上限

**一鍵開戶(`setupAccount`)**:單筆交易原子完成——授權代理密鑰 + 存入首筆保證金 + 註冊推薦碼。將傳統 DeFi 開戶的 4–6 次錢包彈窗壓縮到 1–2 次(首次另需一次 KUSDT approve)。

### 3.3 交易執行:三條路徑,層層降級

**主路徑——Gasless 簽名訂單(平台代付 Gas)**:
1. 代理密鑰以 **EIP-712** 離線簽署訂單(市場、方向、保證金、倉位、滑點上限、nonce、鏈上時間 deadline)
2. 前端 POST 至平台 Relayer
3. Relayer 呼叫 `executeSignedOrderWithPrice(order, orderSig, price, ts, priceSig)`——**單筆原子交易**內:驗證 Keeper 簽名的即時價格並寫入預言機 → 驗證訂單簽名與 nonce(防重放)→ 檢查滑點上限 → 執行開/平倉
4. Relayer 等待鏈上回執後才回覆前端,確保用戶看到的是真實結果(曾採用樂觀回覆,因「假成功」問題撤回)
5. 用戶全程:**零 Gas、零彈窗**

**備援路徑——鏈上一鍵**:Relayer 不可用時,代理密鑰自行上鏈提交兩步驟訂單(需代理有少量 Gas)。

**傳統路徑——兩步驟訂單(反搶跑的基礎設計)**:
1. 用戶提交「意向」(`createIncreaseRequest`),附執行費(0.001 KUB)與可接受價格上限
2. Keeper 在**下一次新鮮報價後**執行(`executeRequest`),以執行當刻價格成交
3. 超過 300 秒未執行的請求只可取消退款;用戶 60 秒後可自行取消

### 3.4 執行時定價(Execution-time Pricing)——反搶跑核心

所有成交(含 Gasless 路徑)都以**執行當刻的新鮮預言機價**結算,而非下單時的顯示價:

- 消除「看到滯後價格再下單」的套利空間——**價格滯後本身無法被利用**,因為成交永遠用執行時市價
- 用戶以滑點上限(0.3% / 0.5% / 1% / 不限)保護自己;市價平倉不設價格上限(由簽名價 20% 偏離上限保護),避免行情波動時平倉被無謂拒絕
- 曾評估「鏈上價格夠新即跳過推價」的提速方案,因會重新引入可利用的滯後窗口而**明確棄用**——寧可多花一個區塊,不留套利面

### 3.5 預言機:多源中位數 + 按需報價(Pull Oracle)

Bitkub Chain 沒有 Chainlink / Pyth,我們自建了兩層預言機:

**價格來源(鏈下聚合)**:
- BTC / ETH:**Binance、OKX、Bybit、Bitkub 四所中位數**——任何單一交易所被操縱、宕機或報出異常值,都會被中位數自然剔除;每個源獨立容錯,單源失敗不影響整體
- KUB:Bitkub Exchange(全球唯一現貨市場)`KUB/THB ÷ USDT/THB` 換算為 USD
- Keeper 每 5 秒鏈下刷新(零 Gas),前端行情、盈虧顯示即時跟隨

**上鏈方式(兩種,各有防護)**:
| 方式 | 用途 | 防護 |
|---|---|---|
| `setPrices`(推送) | 有待執行訂單時報價 | 僅白名單 Keeper;**單次更新偏離上限 5%**,大幅波動須分步逼近 |
| `applySignedPrice`(拉取,Pyth 式) | 隨交易/清算捆綁提交 | **任何人可提交**,合約驗證:簽名者屬 Keeper 白名單、時效 ≤30 秒、偏離 ≤20%、時間戳單調遞增(防舊價重放) |

**按需報價,閒置零 Gas**(成本設計的關鍵):
- 只在「有待執行的兩步驟訂單」時上鏈推價
- 持倉監控、TP/SL 判斷、清算判斷全部**鏈下**以即時 CEX 價進行,到達條件才推價/捆綁簽名價執行
- 無人交易時鏈上零成本;曾實測一個閒置倉位在舊設計下每日耗費約 16 KUB 心跳報價,重構後歸零

### 3.6 清算:鏈下監控 + 簽名價原子清算

- Keeper 每輪以即時 CEX 價鏈下重算每個倉位的償付能力(入場價、保證金、未實現盈虧 vs 維持保證金 1%),**免 Gas**
- 倉位資不抵債時呼叫 `liquidateWithSignedPrice`——**清算與新鮮簽名價捆綁在同一筆原子交易**:即使價格跳空(gap),也以最新市價即時清算,將壞帳壓到最小
- 合約是最終裁判:鏈下估算僅決定「何時嘗試」,清算是否成立由合約以應用後的價格重新驗證
- 清算函數 **permissionless**(驗證簽名價而非呼叫者)——清算錢包無需任何授權,只需 Gas;未來可開放第三方清算人網絡,水平擴容零合約改動

### 3.7 費用結構、VIP 等級與推薦系統

**費用一覽**:
| 項目 | 參數 | 去向 |
|---|---|---|
| 開/平倉費 | 名義倉位的 0.03% | 70% XPLP 池 / 30% 協議金庫 |
| 持倉借貸費 | 按小時累計,費率隨該方向池使用率浮動 | 100% XPLP 池 |
| 急速平倉費 | 開倉 30 秒內平倉加收 0.01% | 100% XPLP 池(反刷量/反高頻套利) |
| 清算罰金 | 清算時從剩餘保證金扣取 | XPLP 池 |
| 執行費 | 0.001 KUB(兩步驟/觸發單) | 補償 Keeper Gas;取消時退回 |

**VIP 等級**(鏈上逐日桶記錄,14 天滾動加權交易量自動計算,無需申請):
| 等級 | 14 天交易量 | 手續費折扣 |
|---|---|---|
| VIP 1 | ≥ $50,000 | 10% |
| VIP 2 | ≥ $250,000 | 25% |
| VIP 3 | ≥ $1,000,000 | 50% |

前端即時顯示當前等級、實際費率(劃線原價 + 折後價)與升級進度。

**推薦系統(全鏈上,XKubReferral)**:
- 開戶時自動生成推薦碼(亦可自訂),分享連結 `?ref=CODE` 對方點開自動綁定
- 被推薦人:手續費 **9 折**(鏈上 `discountOf` 與 VIP 折扣疊加計算)
- 推薦人:賺取被推薦人手續費的 **10% 返佣**,鏈上累計、隨時領取
- 推薦頁展示:好友數、預估推薦交易量、累計佣金、逐筆佣金記錄(含鏈上交易連結)

### 3.8 止盈止損(TP/SL 觸發單)

- 對整個倉位設定 TP/SL 價格,**鏈上儲存**(非純前端掛單)——關瀏覽器也有效
- Keeper 鏈下以即時價監測;到價時先推該市場新鮮價,再呼叫 `executeTrigger` 以市價全平,平倉款回到交易餘額
- 設定時預付 0.001 KUB 執行費(取消退回);前端提供以「收益率 %」快捷設定(TP +25%~+300%、SL −10%~−70%)並即時試算盈虧

### 3.9 風險控制參數(全部合約層強制)

| 控制 | 設定 | 目的 |

| 槓桿上限 | BTC 40x / ETH 25x / KUB 10x | 按市場深度分級 |

| 反對鎖(anti-straddle) | 同帳戶同市場禁止同時持多空 | 防自成交刷返佣/交易量 |

| 最低保證金 | $10(扣開倉費後計) | 防塵埃倉位膨脹清算成本 |
| 維持保證金率 | 名義值 1% | 清算線 |
| 價格時效 | 成交用價 ≤300 秒;簽名價 ≤30 秒 | 杜絕舊價成交 |

## 四、鏈下基礎設施詳解

### 4.1 Keeper / Relayer(常駐服務)

單一輕量服務(獨立純 ethers 建置,記憶體 ~73MB,可跑在 512MB 入門主機)承擔四項職責,**各職責使用獨立錢包與獨立 nonce 通道**:

| 錢包 | 職責 | 鏈上權限 | 設計要點 |
|---|---|---|---|
| W1 報價 | `setPrices` + EIP-712 簽價 | oracle keeper | 簽名與提交分離:W1 簽的價可由任何錢包提交 |
| W2 交易中繼 | 執行簽名訂單/兩步驟/觸發單 | router keeper | **支援錢包池**(逗號分隔多鑰,輪替派單):吞吐 ×N;單筆卡鏈只阻塞該錢包,其餘通道照行 |
| W4 清算 | `liquidateWithSignedPrice` | **無需授權** | 崩盤時清算潮與用戶平倉潮同時爆發,獨立通道互不搶道 |
| W3 水龍頭 | 派發測試幣 | 無需授權 | 詳見 4.2 |

工程細節:
- **每錢包交易串行鎖**:同錢包並發送出會撞 nonce 導致靜默失敗(實戰踩坑後修復),送出階段逐一排隊、等待回執並行
- Gas 價格夾取 60–100 gwei(鏈上 `eth_gasPrice` 虛報 ~185,實測 100 穩定上鏈),所有鏈上動作成本即時減半
- `/errors` 診斷端點:環形緩衝記錄最近 50 筆中繼/水龍頭錯誤,用戶回報問題時可即時定位,無需登入主機

### 4.2 測試幣水龍頭(降低試用門檻的關鍵)

Email 用戶(嵌入式錢包)初始既無 KUB(付不起開戶 Gas)也無 KUSDT(無保證金);官方水龍頭有 Cloudflare 驗證碼無法整合。自建水龍頭直接解決:

- 一鍵領取:**0.05 KUB(開戶 Gas,精算過足夠 approve + setupAccount)+ 10,000 測試 KUSDT**
- KUSDT 從水龍頭錢包預鑄庫存派發(現庫存 1,000 萬,低於閾值自動補鑄)——**不限 IP**,測試資產無稀缺性
- KUB(真正稀缺的 Gas)三重防濫用:每地址一次、每 IP 每 6 小時 3 次(照顧手機 NAT / 辦公室共享 IP)、水龍頭錢包保留額保護(永不派到影響自身運作)
- 前端「領取測試幣」彈窗提供三重保障:平台水龍頭 / **自行由代幣合約 mint**(水龍頭沒貨也可用)/ 官方水龍頭連結
- 開戶流程自動整合:偵測到新錢包缺幣時自動代領,表單在資金到帳前鎖定並引導領取

### 4.3 前端(Next.js 16 / Vercel)

**交易體驗**:
- 實時行情:**Bitkub WebSocket**(單一共享連線,THB→USD 換算)秒級跳動;斷線自動降級 REST
- 倉位盈虧實時重算:鏈上盈虧(含借貸費)為底,疊加即時價差——顯示即時,結算仍以鏈上為準
- 自繪 K 線圖(TradingView Lightweight Charts):Bitkub 數據、USD 計價,疊加**進場價 / TP/SL / 掛單**價位線,各自可開關(設定持久化)
- 下單面板:保證金/倉位大小雙模式、幣本位/U 本位切換、按「可用資金 × 槓桿」動態生成快捷注碼、強平價即時試算、開倉費含 VIP 折扣展示
- 下單前置驗證與合約同構(最低保證金含開倉費計算),高槓桿小注碼在前端即獲明確提示而非鏈上回滾
- 樂觀 UI:下單即顯示「確認中」佔位倉位,鏈上確認後轉正;平倉行變暗鎖定防連點;倉位表固定欄寬,價格跳動不抖行

**可靠性工程(KUB Chain 專屬適配)**:
- KUB 無 EIP-1559 → 所有錢包交易強制 Legacy(Type-0)+ 明確 gasLimit(否則節點靜默丟棄 / MetaMask 高估報價卡死)
- 公共 RPC 瀏覽器 CORS 不穩 → `/api/rpc` 同源代理 + **雙上游容錯**(bitkub 主 / thirdweb 備,各 8 秒超時)
- 錢包網絡自動偵測 + 自動切鏈/加鏈(`wallet_addEthereumChain`),新用戶零手動配置;交易鎖定 chainId 杜絕錯鏈發送
- Gasless 提交失敗自動重試;重覆提交(409)正確識別為「已受理」不誤報失敗

**其他頁面**:資產頁(出入金、VIP 卡、逐筆歷史)、Earn 頁(XPLP 存取、金庫狀態、風險說明)、推薦頁(連結生成、佣金明細);交易歷史逐筆連結 KubScan;完整中英雙語(語言切換持久化);明暗雙主題。

## 五、安全設計

### 5.1 開發過程中主動識別並修復的漏洞

| 漏洞類別 | 對照歷史事故 | 修復 |
|---|---|---|
| 首存通脹攻擊 | 多起 ERC-4626 類金庫事故 | 首存銷毀死份額(MINIMUM_LIQUIDITY) |
| 冷靜期騷擾(griefing) | — | 提款計時按存款金額加權,小額存款無法重置大額計時 |
| 金庫重入 | GMX v1 2025 年 7 月 $42M 重入事故(同類架構) | 池所有資金路徑 ReentrancyGuard |
| 預言機操縱 | Mango Markets 等 | 多所中位數 + 推送 5% / 簽名 20% 偏離上限 + 30 秒時效 + 時間戳單調 |
| 帳戶對鎖抽池 | — | 同帳戶反對鎖 + 盈利上限 300% |
| 簽名重放 | — | EIP-712 域分離 + 鏈上 nonce + deadline(以鏈上時間計) |
| Keeper 供 Gas 被抽乾 | — | 曾評估「平台替代理錢包充 Gas」方案,因可被批量帳戶抽乾而**棄用**,改為 Relayer 代付(資金永不落用戶控制的地址) |

### 5.2 縱深防禦結構

- **權限分層**:Admin(參數治理)/ Keeper(報價+執行,白名單)/ Agent(僅交易)/ 用戶(資金主權)四層,逐函數最小權限
- **角色錢包分離**:報價、中繼、清算、水龍頭獨立密鑰——單鑰洩漏影響範圍受限;清算與水龍頭錢包本身無任何特權
- **合約是最終裁判**:所有鏈下計算(清算判斷、觸發判斷、前端驗證)只決定「何時行動」,成立與否一律由合約以鏈上狀態重新驗證
- **誠實聲明**:合約未經第三方審計(已列為 mainnet 前置條件);測試網明確標示「資金為測試幣」

### 5.3 測試覆蓋

**74 項合約測試全數通過**,分七套:核心倉位引擎(開平倉/盈虧/借貸費/清算/池償付)、代理授權、推薦系統、費率與 VIP、觸發單、Gasless 簽名訂單(含偽造簽名與重放拒絕)、簽名價格(含過期/偏離/非 Keeper 簽名拒絕)。另在測試網完成多輪端到端實測(含以全新錢包模擬真實新用戶的完整旅程)。

5. **工程沉澱回饋生態**:針對 KUB Chain 的 Legacy 交易、RPC 容錯、錢包適配、自建預言機等經驗全部開源,為後續開發者省去大量踩坑成本
